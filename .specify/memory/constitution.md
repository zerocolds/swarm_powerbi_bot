# swarm_powerbi_bot Constitution

## Core Principles

### I. Spec-Driven Development (NON-NEGOTIABLE)
Любое изменение в кодовой базе начинается со спецификации (`spec.md`).
Без утверждённой спецификации реализация запрещена.
Цикл: specify → clarify → plan → critique → tasks → implement → review → verify → adversarial → merge.

### II. Agent-Only Code (NON-NEGOTIABLE)
Код пишется только AI-агентами через SDD-пайплайн.
Ручные правки в обход spec-kit запрещены.
Все изменения должны быть трассируемы к спецификации и задачам.
Hotfix-ветки допускаются только для P0-инцидентов с последующей ретроспективной спецификацией.

### III. Test-First (NON-NEGOTIABLE)
Код без тестов не считается готовым.
Порог покрытия: 60% (начальный), целевой 80%.
Каждый User Scenario в спецификации должен иметь соответствующий тест.
pytest + pytest-asyncio — единственный тестовый фреймворк.

### IV. Multi-Agent Review Gate (NON-NEGOTIABLE)
Перед adversarial-этапом обязательно прохождение полного цикла speckit-ревью:
1. `speckit.review.run` — код, тесты, типы, ошибки, комментарии, упрощение
2. `speckit.staff-review.run` — архитектура, безопасность, производительность, соответствие спеке
3. `speckit.verify.run` — верификация требований и сценариев
4. `speckit.verify-tasks.run` — проверка выполнения задач
5. `speckit.critique.run` — продуктовая и инженерная критика (на этапе плана)
Вердикт "CHANGES REQUIRED" или "RETHINK" блокирует переход на следующий этап.

**Все ревью-агенты запускаются ТОЛЬКО из `.specify/extensions/`.**
Сторонние плагины (metaswarm, внешние MCP-серверы и т.д.) запрещены в SDD-пайплайне.
Промпты для ревью берутся из `commands/*.md` соответствующего расширения.

### V. Adversarial Review Gate (NON-NEGOTIABLE)
Кросс-модельное ревью внешними AI-инструментами — финальный гейт перед мержем.
Запускается ПОСЛЕ прохождения всех speckit-ревью (принцип IV).

**Реализация**:
- Скрипт: `.specify/scripts/adversarial-review.sh`
- Команда: `/speckit.adversarial-review` (`.claude/commands/speckit.adversarial-review.md`)
- Замечания: `.maqa/adversarial-findings.md`

**Участники**: DeepSeek V3.2 (Ollama cloud) и Codex CLI — запускаются параллельно, изолированно друг от друга.
Каждый получает: спецификацию из `.specify/specs/*/spec.md` + diff `main...HEAD`.
Каждый возвращает вердикт: **PASS** или **FAIL** + список замечаний с severity.

**Протокол при FAIL**:
1. Замечания сохраняются в `.maqa/adversarial-findings.md`.
2. Feature-агент (Claude Sonnet) получает замечания и исправляет код.
3. Повторный прогон ТОЛЬКО того ревьюера, который дал FAIL.
4. Максимум **3 раунда** исправлений.
5. Если после 3 раундов вердикт всё ещё FAIL — **эскалация на человека** с полным логом замечаний всех раундов. Авто-мерж запрещён.

**Протокол при PASS**: PASS от обоих ревьюеров → мерж разрешён.

### VI. Adversarial Diversity (NON-NEGOTIABLE)
Качество кода обеспечивается разнообразием ревьюеров — ни один провайдер не ревьюирует только свой код.
- Claude (Opus) пишет и проводит внутреннее QA через MAQA.
- DeepSeek V3.2 (Ollama cloud) и Codex (GPT) проводят независимое adversarial-ревью.
- Каждый ревьюер работает без доступа к замечаниям другого (параллельно, изолированно).
- Совпадение замечаний от разных провайдеров повышает приоритет исправления.

### VII. MAQA для параллельной реализации
Для фич с ≥3 независимыми задачами используется `speckit.maqa.coordinator`.
Максимум 3 параллельных агента (настраивается в `maqa-config.yml`).
Каждый агент работает в изолированном worktree.

**Модели MAQA-агентов** (только Claude):
- **coordinator** — Haiku (оркестрация, маршрутизация)
- **feature** — Sonnet (реализация кода)
- **qa** — Opus (глубокое ревью, adversarial-промпт)

QA-агент (Opus) проверяет каждую ветку перед слиянием в feature-ветку.

### VIII. Документация — часть Definition of Done
Документация обновляется одновременно с кодом в том же PR.
Изменения в API, моделях, конфигурации или процедурах — обязательно отражаются в `docs/`.
CLAUDE.md обновляется при изменении структуры проекта или команд.

### IX. Типизация и качество кода
Python 3.11+, строгая типизация (type hints обязательны для публичных API).
ruff — единственный линтер и форматтер.
Без mypy (по решению команды), но type hints проверяются на ревью.
SQL-инъекции предотвращаются whitelist-валидацией процедур.

### X. Секреты и конфиденциальные данные (NON-NEGOTIABLE)
Токены, строки подключения, API-ключи и любые секреты никогда не коммитятся в репозиторий.
Все секреты хранятся в `.env` (локально) или в GitHub Secrets / переменных окружения (CI/CD).
Файл `.env.example` содержит шаблон всех необходимых переменных без значений.
`.env` включён в `.gitignore`. Любой PR, содержащий секреты, блокируется.

## Процесс разработки

### SDD-цикл (полный)
```
 1. specify          Написать спецификацию
 2. clarify          Уточнить с заказчиком
 3. plan             Составить план реализации
 4. critique         Продуктовая + инженерная критика плана
 5. tasks            Разбить на задачи
 6. MAQA/implement   Параллельная реализация (coordinator→feature→qa)
 7. review           Код-ревью (6 агентов)
 8. staff-review     Архитектурное ревью (5 проходов)
 9. verify           Верификация требований и сценариев
10. verify-tasks     Проверка выполнения задач
11. adversarial      DeepSeek ∥ Codex (≤3 раундов → эскалация)
12. merge            PR и мерж в main
```

### Quality Gates
| # | Гейт | Инструмент | Блокирует | При FAIL |
|---|---|---|---|---|
| 1 | Критика плана | `critique` | Реализацию | Возврат к plan/spec |
| 2 | MAQA QA (Opus) | `maqa.qa` | Слияние worktree | Возврат feature-агенту |
| 3 | Код-ревью | `review` (6 агентов) | Мерж | Исправление → повтор |
| 4 | Staff-ревью | `staff-review` (5 проходов) | Мерж | Исправление → повтор |
| 5 | Верификация | `verify` + `verify-tasks` | Мерж | Исправление → повтор |
| 6 | Тесты | `pytest -q` (≥60%) | Мерж | Исправление → повтор |
| 7 | Линтинг | `ruff check && ruff format --check` | Мерж | Автоформат → повтор |
| 8 | Adversarial | DeepSeek V3.2 ∥ Codex CLI | Мерж | До 3 раундов → эскалация |
| 9 | Проверка секретов | `.env` не в коммите | Мерж | Блокер, ручная очистка |
| 10 | Docker build | `docker build` | Деплой | Исправление → повтор |

### Adversarial Review — детальный flow
```
speckit-ревью PASSED
  │
  ├──► DeepSeek V3.2 (параллельно) ──► PASS / FAIL + замечания
  │                                       │
  └──► Codex CLI     (параллельно) ──► PASS / FAIL + замечания
                                       │
                    ┌──────────────────┘
                    ▼
            Оба PASS? ──yes──► MERGE разрешён
                │
               no
                │
                ▼
         Собрать FAIL-замечания
                │
                ▼
         Feature-агент (Sonnet) исправляет
                │
                ▼
         Повторный прогон ТОЛЬКО failed-ревьюера
                │
                ▼
         PASS? ──yes──► проверить второго ──► оба PASS? ──► MERGE
                │
               no
                │
                ▼
         Раунд < 3? ──yes──► повтор исправления
                │
               no
                │
                ▼
         ЭСКАЛАЦИЯ НА ЧЕЛОВЕКА
         (полный лог всех раундов)
```

### Соглашения
- Ветки: `feature/NNN-short-name` (sequential numbering через spec-kit)
- Коммиты: на русском, через `speckit.git.commit`
- PR: один PR на одну спецификацию

## Governance

Конституция имеет приоритет над всеми другими практиками и соглашениями.
Изменения в конституции требуют: документации причины, ревью, и обновления версии.
Все агенты и ревьюеры обязаны проверять соответствие конституции.
Нарушение NON-NEGOTIABLE принципов является блокером для любого PR.
