# Phase 0 Research — Парсинг bare-months

**Date**: 2026-04-17
**Feature**: [spec.md](./spec.md)

Все NEEDS CLARIFICATION из Technical Context разрешены на этапе `/speckit.clarify` (две сессии: падежные формы март/август, приоритет bare-month). Ниже — подтверждение ключевых технических решений.

---

## R-001. Порядок проверок в `extract_date_params`

**Decision**: bare-month проверяется в следующем порядке:

1. `_RE_MONTH` — «за <месяц> [<год>]» (обратная совместимость)
2. `_RE_RANGE` — «с X по Y <месяц> [<год>]»
3. «вчера» (абсолютный якорь)
4. «сегодн» (абсолютный якорь)
5. **`_RE_MONTH_BARE` — bare-month (новая ветка)** ← ВСТАВКА
6. «недел» → last 7 days
7. «месяц»/«меся» → last 30 days
8. «квартал» → last 90 days
9. «полугод»/«полгод» → last 180 days
10. «год» (кроме «новый год») → last 365 days
11. default: last 30 days

**Rationale**: Clarification Q2 зафиксировал — bare-month имеет приоритет над keyword-периодами. Вставка между абсолютными якорями («вчера»/«сегодня», которые остаются как контекст на сегодняшний день) и общими окнами обеспечивает корректный результат для смешанных запросов вида «отчёт за месяц март» (→ март) без потери «вчера» (которая должна остаться абсолютным якорем, а не быть перекрытой каким-то bare-month в той же строке — «вчера» не содержит месяцев, так что конфликта нет).

**Alternatives considered**:
- Поставить bare-month в самое начало (до `_RE_MONTH`) — отбросили: нарушит FR-005 (приоритет «за <месяц>»).
- Поставить после keyword-периодов — отбросили: Clarification Q2 прямо требует обратного.

## R-002. Расширение regex `_RE_MONTH_BARE` падежами «март»/«август»

**Decision**:

- Текущие паттерны: `март\b`, `август\b`.
- Новые: `март[аеуом]?\b`, `август[аеуом]?\b` — покрывают родительный (марта/августа), дательный (марту/августу), предложный (марте/августе), творительный (мартом/августом). Для творительного падежа используется суффикс `ом`, поэтому класс становится `[аеуом]?` — но регекс с одним символьным классом не даст двухсимвольного суффикса.

  **Уточнение**: двухсимвольное окончание «ом» нельзя выразить одним символьным классом. Используем альтернацию:
  - `март(?:а|у|е|ом)?\b`
  - `август(?:а|у|е|ом)?\b`

  Это сохраняет читаемость и точно покрывает все 5 форм (включая именительный через `?`).

- Существующие паттерны для остальных месяцев вида `январ[яеюиь]?` покрывают именительный/родительный/дательный/предложный (января, январе, январю, январи), но НЕ покрывают творительный (январём/январем). Решение: **расширить аналогично и их**, чтобы поведение было единым:
  - `январ(?:я|е|ю|и|ём|ем)?\b` — расширение до творительного падежа.
  - Для упрощения и устойчивости (ё/е) принимаем оба варианта написания («январём» и «январем»).

  Это слегка расширяет scope vs изначальный текст спеки, но соответствует принципу единообразия, зафиксированному в Clarification Q1.

**Rationale**: Пользователи в аналитических запросах часто употребляют творительный падеж («отчёт январём», «сравни мартом и апрелем»). Единый подход ко всем месяцам упрощает поддержку.

**Alternatives considered**:
- Оставить старые паттерны и добавить только `[аеуом]?` для «март»/«август» — отбросили: асимметрия между месяцами усложнит поддержку.
- Полный морфологический разбор через `pymorphy2` — отбросили: новая тяжёлая зависимость ради узкого кейса нарушает принципы простоты и минимальных правок.

## R-003. Word-boundary `\b` и кириллица в Python `re`

**Decision**: `\b` в Python `re` с Unicode-строками корректно работает для кириллических слов (по умолчанию с Python 3.x включён `re.UNICODE`). Использование `\b` обеспечивает защиту от ложных срабатываний на составных словах («мартышка», «августовский», «майонез»).

**Rationale**: Проверено для `_RE_MONTH_BARE` уже сейчас; расширение не меняет семантики.

**Alternatives considered**:
- Явные lookaround'ы `(?<![а-яё])...(?![а-яё])` — избыточно, `\b` достаточно.

## R-004. Сохранение поведения `has_period_hint()`

**Decision**: `has_period_hint()` уже использует `_RE_MONTH_BARE` в проверке. Расширение regex автоматически покрывает новые формы, новые тесты (`test_period_hint.py`) должны это подтвердить. Никаких изменений в саму функцию `has_period_hint()` не требуется.

**Rationale**: FR-009 требует нулевой регрессии — проверяется тестами.

---

## R-005. Сравнение «март и апрель» в single-query fallback

**Decision**: `_RE_MONTH_BARE.search()` возвращает первое совпадение по позиции. Этого достаточно для FR-006. Отдельная логика сбора всех месяцев не требуется, т.к. comparison обрабатывается multi-query планировщиком, а single-query — лишь fallback.

**Rationale**: минимизируем scope, не дублируем логику comparison.

**Alternatives considered**:
- Возвращать range [первый_месяц..последний_месяц] — отбросили: нарушит семантику single-query fallback (функция должна вернуть ровно один период), плюс это продуктовое решение, не входящее в фикс bug-а.

## R-006. Покрытие тестами

**Decision**: минимальный набор новых тестов в `tests/test_period_detection.py`:

- Именительный падеж: «выручка март 2026», «услуги февраль 2025», …, «отчёт декабрь 2026» (12 тестов — параметризованные через `pytest.mark.parametrize`, **всегда с явным годом** — см. ниже).
- Родительный/дательный/предложный/творительный падеж: «итоги марта 2026», «в апреле 2026», «мартом 2026», «августом 2026» (≥4 теста, год явно).
- Приоритет «за»: «выручка за март 2025» → `2025-03-01..2025-03-31` (регрессия существующего).
- Приоритет bare-month над keyword: «отчёт за месяц март 2026» → март 2026; «за неделю июня 2026» → июнь 2026; «за год январь 2025» → январь 2025.
- Приоритет «за <месяц>» над bare-month другого месяца: «за январь 2025 февраль» → январь 2025.
- Несколько месяцев: «сравни март 2026 и апрель 2026» → март 2026 (первый).
- **Negative — составные слова (≥15 кейсов)**: ровно по 1 составному слову на каждый из 12 месяцев из `_MONTH_MAP` (например: январский, февральский, мартышка, апрельский, майонез, июньский, июльский, августовский, сентябрьский, октябрятский, ноябрьский, декабрьский) + 3 дополнительных (например: «мартини», «маяковский», «подмайский»). Все → default last-30-days, `has_period_hint` = False.
- Год указан: «февраль 2025» → `2025-02-01..2025-02-28`; «февраль 2024» → `2024-02-01..2024-02-29` (високосный).
- Регрессионные: «за неделю», «за квартал», «вчера», «сегодня» — не должны сломаться.
- **Без явного года** (для подтверждения поведения дефолта текущего года): 1–2 теста с `monkeypatch` на `sql_client.date` с фиксированной датой 2026-04-17.
- **Regex-страж против ReDoS** (см. R-007): `extract_date_params("а" * 10_000)` и `has_period_hint("а" * 10_000)` завершаются за < 10 мс, не падают с `RecursionError`.

Плюс в `tests/test_period_hint.py` добавить параметризованный тест для bare-month форм (именительный + творительный) для всех 12 месяцев, чтобы подтвердить FR-009.

**Rationale**: SC-003 (0 ложных срабатываний — ≥15 negative-тестов гарантируют), SC-004 (покрытие ≥80%), SC-002 (0 регрессий), стабильность тестов (явный год исключает flakiness на стыке лет, см. plan.md).

## R-007. ReDoS-инвариант для `_RE_MONTH_BARE`

**Decision**: Новый regex `_RE_MONTH_BARE` обязан удовлетворять следующим правилам, которые делают catastrophic backtracking невозможным:

1. Все альтернативы — плоские, без вложенных квантификаторов (`*`, `+`, `{m,n}`).
2. Альтернативы в non-capturing группе (`(?:...|...)`) имеют различимые префиксы (разные стемы месяцев); общих префиксов с квантификаторами нет.
3. Для падежных окончаний используется только опциональная группа `(?:а|е|у|и|ём|ем|я|ю|ь)?` без повторений — один символ или двухсимвольное окончание (`ём`/`ем`), без `*` или `+`.

**Проверка**: страж-тест (см. R-006) на входе `"а" * 10_000` — выполняется за <10 мс. Дополнительно фиксируем инвариант в [contracts/extract_date_params.md](./contracts/extract_date_params.md).

**Rationale**: Даже при простом плоском regex расширение альтернатив со временем может ввести общий префикс (например, при добавлении нового месяца). Инвариант + страж-тест защищают от регрессий.

**Alternatives considered**:
- Переход на предварительно токенизированный lookup по `_MONTH_MAP.startswith()` — безопаснее, но переписывает существующий стиль. Отвергнуто: превышает scope фикса.

## R-008. Structured logging для SC-006

**Decision**: В `extract_date_params` добавить ровно одно структурированное сообщение DEBUG-уровня через существующий логгер `logger` (`logging.getLogger(__name__)`, уже инициализирован в файле):

```
logger.debug(
    "period_extracted",
    extra={
        "strategy": "bare_month|za_month|range|absolute|keyword|default",
        "date_from": str(date_from),
        "date_to": str(date_to),
        "question_len": len(question),
    },
)
```

- Сообщение пишется ровно 1 раз за вызов, перед `return`.
- Поле `question` **не логируется** целиком (избегаем PII в логах); вместо этого — длина и стратегия.
- SC-006 измеряется grep'ом по `strategy="default"` в сочетании с наличием bare-month в исходном вопросе (поле `question_len` — для быстрого санити-чек; полный текст уже пишется `query_logger`-ом на другом уровне, см. FR-009 наследство из issue #17 — проверка в продакшн-лог оркестратора).

**Rationale**: FR-011 + SC-006 требуют машинно-измеримого сигнала. DEBUG-уровень не нарушает production logging volume (если не включён явно), но доступен для выборочного включения при QA.

**Alternatives considered**:
- `INFO`-уровень — отвергнуто: засоряет продовый лог при нормальной работе.
- Отдельный метрический счётчик (Prometheus) — избыточно для scope фикса; DEBUG-лог легко агрегируется пост-фактум.
