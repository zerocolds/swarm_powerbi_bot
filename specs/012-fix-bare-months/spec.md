# Feature Specification: Парсинг bare-months в extract_date_params

**Feature Branch**: `012-fix-bare-months`
**Created**: 2026-04-17
**Status**: Draft
**Input**: User description: "extract_date_params() не парсит bare months (январь, март без «за»)"
**Tracks**: GitHub issue [#17](https://github.com/k-p-i-bi/swarm_powerbi_bot/issues/17)

## Clarifications

### Session 2026-04-17

- Q: Входят ли в scope падежные формы для «март» и «август» (марта, марту, мартом, марте / августа, августу, августом, августе)? → A: Да, включить — расширить распознаваемые формы для «март» и «август» аналогично остальным месяцам, покрыть падежи тестами.
- Q: Какой приоритет должен быть у bare-month относительно ключевых слов периода («неделя», «месяц», «квартал», «год», «полугодие»)? → A: Bare-month побеждает keyword-период: «за месяц март» → март; «за год январь 2025» → январь 2025.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Запрос выручки за конкретный месяц без предлога (Priority: P1)

Пользователь-аналитик формулирует вопрос обиходно: «выручка март», «услуги февраль», «новые клиенты январь 2026». Он ожидает, что бот поймёт указанный месяц и вернёт данные именно за него, а не за произвольные последние 30 дней.

**Why this priority**: Это основной кейс, из-за которого заведён баг. Сейчас пользователь получает цифры, которые не соответствуют его вопросу, что подрывает доверие к боту и искажает аналитику.

**Independent Test**: Полностью проверяется на `extract_date_params("выручка март")` и аналогичных строках без предлога «за» — функция должна вернуть период (1-е—последний день) запрошенного месяца, а не скользящее окно.

**Acceptance Scenarios**:

1. **Given** текущий год 2026, **When** пользователь спрашивает «выручка март», **Then** система использует период `2026-03-01`..`2026-03-31`.
2. **Given** запрос «услуги февраль 2025», **When** извлекаются параметры, **Then** система использует период `2025-02-01`..`2025-02-28`.
3. **Given** запрос «новые клиенты январь», **When** текущий год 2026, **Then** используется период `2026-01-01`..`2026-01-31`.
4. **Given** запрос «итоги декабря», **When** извлекаются параметры, **Then** используется период `2026-12-01`..`2026-12-31` (падежные формы распознаются).

---

### User Story 2 — Сохранение существующего поведения с предлогом «за» (Priority: P1)

Пользователи, привыкшие писать «выручка за март», «данные за январь 2025», должны получать тот же результат, что и раньше. Новая логика bare-months не должна перехватывать или ломать эти запросы.

**Why this priority**: Регрессия на работающем синтаксисе неприемлема — это основной канал пользовательских запросов за прошедшие релизы.

**Independent Test**: Проверяется на строках «за март», «за январь 2025» и сравнении результата до/после изменения — он должен совпадать.

**Acceptance Scenarios**:

1. **Given** запрос «выручка за март», **When** извлекаются параметры, **Then** используется период `2026-03-01`..`2026-03-31` (поведение не изменилось).
2. **Given** запрос «за январь 2025», **When** извлекаются параметры, **Then** используется период `2025-01-01`..`2025-01-31`.

---

### User Story 3 — Сравнение двух месяцев в одном запросе (Priority: P2)

Пользователь пишет «сравни март и апрель» — это comparison-запрос. Оркестратор уже маршрутизирует такие запросы на multi-query планировщик, однако первичная функция извлечения периода должна корректно определить хотя бы первый упомянутый месяц, чтобы не падать на default last-30-days, если multi-query не применён.

**Why this priority**: Это граничный случай, важный для устойчивости fallback-а, но основную работу по comparison делает отдельный planner. Приоритет ниже P1.

**Independent Test**: `extract_date_params("сравни март и апрель")` должен вернуть период марта (первый найденный месяц), а не default last-30-days.

**Acceptance Scenarios**:

1. **Given** запрос «сравни март и апрель», **When** извлекаются параметры single-query fallback-ом, **Then** используется период первого упомянутого месяца (март).
2. **Given** запрос «разница между январём и февралём 2025», **When** single-query fallback, **Then** используется период первого упомянутого месяца (январь 2025).

---

### Edge Cases

- «май» vs «мая» — обе формы (именительный, родительный) должны распознаваться.
- «март» / «август» в падежных формах («марта», «марту», «мартом», «марте», «августа», «августу», «августом», «августе») — распознаются наравне с формами других месяцев.
- «новый год» — слово «год» не должно триггерить парсер bare-month для отсутствующих названий месяца; сочетание «новый год» уже обрабатывается отдельно.
- Год в будущем/прошлом, явно указанный («январь 2030», «март 2019»), — используется указанный год, без проверки на «разумность».
- Год не указан — используется текущий календарный год (на момент вызова функции).
- Название месяца встречается внутри слова (например, «мартышка», «августовский») — парсер не должен ложно срабатывать на такие вхождения.
- Приоритет: если в строке есть и «за <месяц>», и bare-month вида «<месяц>», приоритет у формы с «за» (обратная совместимость). Пример: «за январь февраль» → январь (bare-month «февраль» игнорируется).
- Если в строке два и более bare-month (сравнение), выбирается первый по позиции.
- Приоритет bare-month над keyword-периодами: если в запросе одновременно встречаются конкретный месяц и общее ключевое слово периода («месяц», «неделя», «квартал», «год», «полугодие»), побеждает bare-month. Примеры: «отчёт за месяц март» → март; «данные за год январь 2025» → январь 2025; «за неделю июня» → июнь (целиком).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST распознавать название месяца в русскоязычном запросе без обязательного предлога «за» (bare-month) во всех основных падежных формах (именительный, родительный, дательный, творительный, предложный) — включая «март»/«август» в падежных формах (марта, марту, мартом, марте, августа, августу, августом, августе).
- **FR-002**: При распознавании bare-month система MUST возвращать период «первый день месяца — последний день месяца» в соответствующем календарном году.
- **FR-003**: Система MUST использовать явно указанный в запросе год (`<месяц> <YYYY>`), если он есть; иначе — текущий календарный год.
- **FR-004**: Система MUST сохранять текущее поведение для запросов с предлогом «за <месяц>» — результат не должен отличаться от версии до фикса.
- **FR-005**: Система MUST отдавать приоритет форме с предлогом «за», если в запросе встречаются обе формы.
- **FR-006**: Система MUST выбирать первый по позиции bare-month, если в запросе упомянуто несколько месяцев (случай сравнения, single-query fallback).
- **FR-007**: Система MUST игнорировать вхождения названий месяцев внутри других слов (мартышка, августовский) — распознаются только слова, стоящие на границах.
- **FR-008**: Система MUST отдавать приоритет bare-month над общими ключевыми словами периода («неделя», «месяц», «квартал», «год», «полугодие»): при одновременном присутствии результат соответствует календарному периоду найденного месяца. Абсолютные якоря «вчера» и «сегодня» сохраняют текущее поведение и не перекрываются bare-month.
- **FR-009**: Система MUST сохранить текущее поведение `has_period_hint()` — добавление bare-month парсинга в `extract_date_params` не меняет результаты проверки наличия периода.
- **FR-010**: Система MUST покрыть новое поведение автотестами: bare-month с годом и без, приоритет «за», несколько месяцев, падежные формы, отсутствие ложных срабатываний на составных словах.
- **FR-011**: Система MUST логировать результат извлечения периода структурированным DEBUG-сообщением, включающим поле `strategy` с одним из значений: `bare_month`, `za_month`, `range`, `absolute` (вчера/сегодня), `keyword` (недели/месяц/квартал/полугодие/год), `default`. Сообщение направляется в стандартный логгер модуля.

### Key Entities

- **Запрос пользователя (question)**: свободный текст на русском языке, может содержать название месяца, год, диапазон, ключевые слова периода.
- **Параметры периода (DateFrom/DateTo)**: пара дат, передаваемая в хранимую процедуру MSSQL; должна соответствовать намерению пользователя.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% запросов вида «<метрика> <месяц>» и «<метрика> <месяц> <год>» (без «за») возвращают период запрошенного месяца, а не default last-30-days.
- **SC-002**: 0 регрессий на запросах с «за <месяц>» — все существующие тестовые сценарии, проверяющие этот синтаксис, продолжают проходить без изменений.
- **SC-003**: 0 ложных срабатываний на составных словах, содержащих корень месяца (мартышка, августовский, майонез и т.п.) — подтверждается тестами.
- **SC-004**: Покрытие автотестами нового парсинга ≥ 80% ветвей новой логики (соответствует целевому порогу конституции).
- **SC-005**: Пользователи, сообщавшие о проблеме (issue [#17](https://github.com/k-p-i-bi/swarm_powerbi_bot/issues/17)), подтверждают корректное поведение на своих исходных примерах («выручка март», «сравни март и апрель»).
- **SC-006**: В продакшн-логах за первые 7 дней после релиза доля запросов, в которых есть bare-month в тексте, но стратегия извлечения периода — `default`, составляет 0% (измеряется по структурированному полю `strategy` в DEBUG-логах и/или `query_logger`).

## Assumptions

- Существующий словарь `_MONTH_MAP` и regex `_RE_MONTH_BARE` покрывают все грамматические формы, которые пользователь может ввести; расширять список месяцев не требуется.
- Поддерживаются падежные формы для всех двенадцати месяцев; для «март» и «август» логика дополняется по аналогии с остальными месяцами; редкие книжные/устаревшие формы не учитываются.
- Год за пределами разумного (например, 1800 или 3000) считается корректным вводом и передаётся в SQL как есть — валидация диапазона лет не входит в scope.
- Comparison-сценарий («сравни март и апрель») в основном обрабатывается оркестратором и multi-query планировщиком; изменение касается только fallback-поведения single-query пути.
- Функция вызывается в одном процессе с актуальным системным временем; `today = date.today()` корректен.
- Изменение локализовано в `src/swarm_powerbi_bot/services/sql_client.py`; внешние контракты (сигнатура `extract_date_params`, `has_period_hint`) не меняются.
