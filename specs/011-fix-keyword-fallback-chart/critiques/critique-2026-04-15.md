## Executive Summary

Спецификация хорошо описывает **симптомы** 4 production-багов (#13-#16) и правильно определяет необходимые исправления для `_FIELD_LABELS`, `_HIDDEN_FIELDS`, `_PREFERRED_VALUE` и `has_period_hint()`. Однако **корневое архитектурное допущение неверно**: spec и plan утверждают, что `PlannerAgent.run()` использует keyword scoring как primary path (FR-001: "заменить keyword scoring на LLM-маршрутизацию"). В действительности `run()` уже вызывает `_llm_plan()` → `LLMClient.plan_query()` первым (planner.py:458), и keyword scoring уже является fallback (planner.py:465).

Это означает, что проблема маршрутизации (#13, #16) скорее всего в **качестве LLM-промпта** (`_PLANNER_SYSTEM_PROMPT`), а не в отсутствии LLM-маршрутизации. Или же LLM по каким-то причинам недоступен в production (timeout, no API key), и бот работает на keyword fallback. Без диагностики первопричины план может решать несуществующую проблему.

**Verdict**: ⚠️ **PROCEED WITH UPDATES** — 1 must-address (диагностика реального routing path) + 4 рекомендации.

## Engineering Lens

### 4a. Architecture Soundness

**Оценка: CRITICAL ISSUE**

**`run()` уже имеет LLM-first routing.**

Код planner.py:450-467:
```python
async def run(self, question: UserQuestion) -> Plan:
    # Пробуем LLM-планирование
    llm_result = await self._llm_plan(question)        # ← LLM FIRST
    used_llm = llm_result is not None

    if used_llm:
        query_params = llm_result
        topic = self._derive_topic(query_params, question)
    else:
        # Fallback: keyword-matching
        topic = detect_topic(question.text, ...)       # ← keyword FALLBACK
```

Spec FR-001 ("PlannerAgent MUST использовать LLM как primary маршрутизатор") **уже выполняется** в текущем коде. Plan описывает "перенос LLM-routing из `run_multi()` в `run()`", но `run()` уже это делает.

**Возможные реальные причины багов #13/#16**:
1. `_PLANNER_SYSTEM_PROMPT` направляет только на v2 процедуры (spKDO_Aggregate, spKDO_ClientList, spKDO_CommAgg), но `_USE_V2=False` → v2-to-v1 remapping может терять контекст
2. Ollama недоступен в production → бот падает на keyword fallback
3. LLM возвращает невалидную процедуру → `_llm_plan()` returns None → keyword fallback
4. Промпт не содержит достаточно контекста для различения statistics vs services при вопросах о «выручке»

**Без диагностики реального пути (logs: `planner:llm` vs `planner:keyword` в production) нельзя определить правильное исправление.**

## Verdict

### ⚠️ PROCEED WITH UPDATES

**Must-Address (E1)**: Необходимо диагностировать реальную причину неправильной маршрутизации:
- Если LLM недоступен (нет API key, timeout) → проблема в конфигурации/инфраструктуре, а не в коде. Фикс: обеспечить доступность Ollama. Keyword fallback можно улучшить, но это не core fix.
- Если LLM доступен, но возвращает неправильный topic → проблема в `_PLANNER_SYSTEM_PROMPT` или `_derive_topic()` remapping. Фикс: улучшить промпт, не дублировать routing.
- Если `_USE_V2=False` и remapping теряет контекст → фикс в `_V2_TO_V1` mapping.

**Рекомендуемые обновления spec/plan**:

1. **spec.md FR-001**: Изменить с "заменить keyword scoring на LLM" на "диагностировать и исправить LLM-маршрутизацию в `run()` — LLM path уже существует, но не работает корректно для вопросов #13/#16"
2. **plan.md Architecture**: Исправить диаграмму "текущее состояние" — `run()` уже LLM-first, не keyword-first
3. **plan.md**: Добавить Phase 0.5 — diagnostic step: проверить logs/config на production path
4. **research.md R1**: Обновить — `run()` не нуждается в "переносе LLM из run_multi()", нуждается в диагностике и улучшении существующего LLM routing
