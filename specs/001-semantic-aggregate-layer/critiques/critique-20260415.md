## Product Lens Findings

### 3a. Problem Validation

Проблема чётко определена: 17 жёстких тем с keyword-scoring не позволяют комбинировать данные, сравнивать периоды и декомпозировать вопросы. Это реальное ограничение для владельцев салонов, которые думают в терминах бизнеса, а не в категориях topic_registry.

**P1** 💡 **Нет количественных данных о частоте "проваленных" вопросов**. Спецификация описывает ограничения as-is, но не приводит статистику: сколько вопросов сейчас не получают ответа, какие типы запросов чаще всего "промахиваются" мимо 17 тем. Эти данные помогли бы приоритизировать агрегаты в Phase 1.

- **Suggestion**: Добавить в Assumptions или отдельным разделом: "На основании [логов/наблюдений] ~X% вопросов не маппятся на существующие темы. Наиболее частые пропуски: сравнения периодов, кросс-доменные запросы." Если данных нет — зафиксировать это как baseline measurement task в Phase 2.

### 3b. User Value Assessment

Все 13 user stories ориентированы на пользовательскую ценность. Фазовая структура (P1→P2→P3) логична — сначала инфраструктура, потом сравнения, потом композиция.

**P2** 💡 **Phase 0 (6 user stories) — это dev-time bootstrap, а не пользовательская ценность**. US-001..US-006 написаны как user stories ("Как аналитик данных, я хочу..."), хотя это одноразовые dev-утилиты. Формат User Story тут избыточен, но acceptance scenarios полезны как checklists. Не блокирует, но создаёт впечатление большего объёма, чем есть.

- **Suggestion**: Переименовать Phase 0 stories в "Bootstrap Tasks" или пометить как dev-internal. User stories оставить для Phase 2-4, где есть реальный конечный пользователь.

**P3** 💡 **Follow-up вопросы упомянуты как edge case, но не специфицированы**. В spec.md US-008 edge case: "Follow-up вопрос ('а по неделям?') — LLM переиспользует контекст предыдущего запроса". Это серьёзная фича (stateful conversation), но не имеет acceptance scenario и не вынесена в отдельный FR.

- **Suggestion**: Либо добавить US/FR для follow-up с acceptance scenario, либо явно вынести в out-of-scope с примечанием "v2". Текущая формулировка создаёт неоднозначность для разработчика.

### 3c. Alternative Approaches

**P4** 🤔 **Нужен ли двухшаговый LLM planning сразу?** При ~20-30 агрегатах и 8 категориях полный каталог может уместиться в context window GLM-5 в одном шаге. Двухшаговый подход добавляет ~1-2 сек латентности и усложняет отладку. Research R-003 упоминает "при росте числа агрегатов", но текущий масштаб невелик.

- **Suggestion**: Вопрос к стейкхолдеру: начать с одношагового planning (проще, быстрее) и переключиться на двухшаговый при >50 агрегатах? Или сразу закладывать двухшаговый для архитектурной готовности?

### 3d. Edge Cases & User Experience

Основные edge cases покрыты: SQL injection, LLM fallback, partial failure, fuzzy-match, неполные периоды. Хорошая проработка.

**P5** 💡 **Не описано поведение при конфликте между keyword-fallback и LLM-планом**. Если LLM работает, но возвращает невалидный план (aggregate_id не из каталога), spec говорит "план отклоняется". Но fallback на TopicRegistry происходит только при "LLM недоступен". Нет сценария: LLM доступен, но выбрал плохо.

- **Suggestion**: Добавить acceptance scenario: "Given LLM вернул aggregate_id не из каталога → Then система делает retry с simplified prompt / falls back на TopicRegistry". Указать стратегию: retry vs fallback vs "не могу ответить".

### 3e. Success Measurement

SC-001..SC-008 — хорошие критерии. SC-007 (≤20% деградация latency) — конкретный и измеримый.

**P6** 🤔 **SC-007 сложно измерить без baseline**. Текущая латентность не зафиксирована. "≤20% увеличения" от чего?

- **Suggestion**: Добавить в Phase 2 задачу: "Замерить baseline latency для 17 типовых вопросов до включения semantic layer". Или зафиксировать текущий baseline в spec.md Assumptions.

## Engineering Lens Findings

### 4a. Architecture Soundness

Архитектура следует устоявшимся паттернам: каталог → registry → validator → executor. Чёткое разделение concerns между CatalogLoader, AggregateRegistry, ParameterValidator. Data flow diagram в data-model.md — наглядный и полный.

**E1** 💡 **Три отдельных модуля (catalog_loader, aggregate_registry, parameter_validator) при малом объёме кода**. Каждый модуль содержит ~50-100 строк. aggregate_registry уже зависит от catalog_loader и parameter_validator. На текущем масштабе достаточно одного модуля `aggregate_registry.py` с тремя функциями.

- **Suggestion**: Объединить catalog_loader + parameter_validator в aggregate_registry.py. Если при реализации объём перевалит за 300 строк — разделить. Три модуля по 50 строк — это premature abstraction для проекта с 14 файлами в services/.

### 4b. Failure Mode Analysis

Хорошее покрытие: partial failure, timeout per query (10s), max 10 queries. LLM fallback на TopicRegistry.

**E2** 💡 **Нет circuit breaker для LLM API**. Если Ollama Cloud деградирует (latency 5-10 сек вместо 1-2), каждый запрос пользователя будет ждать двух LLM-вызовов (шаг 1 + шаг 2) по 5-10 сек, прежде чем fallback. При полной недоступности — два timeout (сколько?). Timeout для LLM calls не указан в spec/plan.

- **Suggestion**: Добавить в plan.md: timeout для LLM calls (например, 5 сек на шаг), и простой circuit breaker: если 3 подряд LLM timeout → переключить на TopicRegistry на 60 сек. Не нужна библиотека — достаточно счётчика в LLMClient.

**E3** 💡 **asyncio.gather для SQL запросов — нет ограничения concurrency**. 10 параллельных SQL запросов к одному серверу могут вызвать connection pool exhaustion или перегрузить MSSQL при нескольких одновременных пользователях.

- **Suggestion**: Использовать `asyncio.Semaphore(5)` или настраиваемый pool size. Для малого числа клиентов (салоны) это не критично, но стоит заложить в plan как config parameter.

### 4c. Security & Privacy Review

Безопасность — сильная сторона спецификации. Whitelist aggregate_id, typed parameters, read-only SQL user, structured logging без данных/токенов. US-012 — детальный и правильный.

**E4** 💡 **Parameter validation: enum whitelist для group_by содержит 11 значений, но нет привязки к конкретным агрегатам**. Не каждый агрегат поддерживает все 11 group_by. Если LLM выберет `group_by: channel` для агрегата, который не поддерживает каналы — runtime error или неожиданные результаты.

- **Suggestion**: В aggregate-catalog.yaml для каждого агрегата указать `allowed_group_by: [master, week, month]`. ParameterValidator проверяет не только по глобальному enum, но и по allowed-списку конкретного агрегата.

### 4d. Performance & Scalability

Materialized views (indexed views) — правильный выбор для MSSQL. Overhead ≤3x — разумное ограничение для салонов. Двухшаговый planning масштабирует каталог.

Производительность адекватна текущему масштабу. Дополнительных замечаний нет.

### 4e. Testing Strategy

12 новых тест-файлов, 77 существующих сохраняются. Покрытие 60%→80%. Тестовый план комплексный.

**E5** 💡 **Нет стратегии тестирования LLM-вызовов (шаг 1 + шаг 2)**. LLM responses недетерминированны. Как тестировать PlannerAgent v2? Mock LLM? Snapshot-тесты? Property-based тесты на контракт вывода?

- **Suggestion**: В plan.md добавить: "Тесты PlannerAgent v2 используют mock LLMClient с фиксированными JSON responses. Тесты проверяют: (1) корректный prompt формируется, (2) JSON response парсится в MultiPlan, (3) невалидный JSON response обрабатывается gracefully. Integration test с реальным LLM — manual, не в CI."

### 4f. Operational Readiness

Structured logging (query_logger.py) — хорошо. JSON формат, rotation 10MB×5.

**E6** 🤔 **Нет метрик для мониторинга health semantic layer**. Как узнать что LLM planning деградировал? Что fallback rate вырос? Что определённый агрегат часто timeout-ит?

- **Suggestion**: Вопрос к стейкхолдеру: нужны ли Prometheus-метрики или достаточно анализа aggregate_queries.log? При малом числе клиентов лог-анализ может быть достаточен. Если да — зафиксировать в plan.md: "мониторинг через парсинг логов, без дополнительной инфраструктуры."

### 4g. Dependencies & Integration Risks

Зависимости хорошо описаны. PBIX dev-time only, чёткое разделение. Existing SP сохраняются.

**E7** 💡 **YAML-каталоги загружаются при старте бота, но нет hot-reload**. При обновлении каталога (Phase 0 повторный bootstrap после изменения PBIX) нужен рестарт бота. При малом числе клиентов это не проблема, но стоит задокументировать.

- **Suggestion**: Добавить в plan.md или quickstart.md: "Обновление каталогов требует рестарта бота. Hot-reload не реализуется (YAGNI при текущем масштабе)."

## Cross-Lens Insights

### X1: Упрощение модульной структуры (Product + Engineering)

Product: меньше модулей = проще код = быстрее реализация.
Engineering: catalog_loader + parameter_validator + aggregate_registry — потенциально один модуль.
**Convergence**: объединение снижает и сложность реализации, и когнитивную нагрузку.

### X2: LLM resilience (Product + Engineering)

Product: пользователь не должен ждать >15 сек из-за деградации LLM.
Engineering: нет timeout для LLM calls и circuit breaker.
**Convergence**: LLM timeout + простой circuit breaker улучшают и UX, и надёжность.

### X3: Follow-up контекст (Product + Engineering)

Product: follow-up — ценная фича для пользователя, но не специфицирована.
Engineering: stateful conversation требует хранения контекста между запросами.
**Convergence**: явно пометить как out-of-scope v1. Это предотвращает и scope creep, и архитектурные сюрпризы.
